# 框架设计、思考方式与客户价值映射

## 1. 这套系统解决什么，不解决什么

我把它当成一个工程系统，不当成聊天产品来设计。核心目标是让输出可接入、可复核、可回退。

它解决的是：

1. 给出可追溯的诊断结论，而不是“听起来合理”的答案。
2. 在信息不完整时主动降级，避免错误代码直接进入生产。
3. 提供稳定接口契约，降低前后端联调成本。

它不解决的是：

1. 全自动替用户执行高风险系统操作。
2. 在证据不足时强行给出确定结论。

## 2. 我做过的关键设计决策

### 2.1 协议先行，不让前端猜状态

- 对外统一同步接口和 SSE。
- SSE 事件只允许 `status/content/final/error` 四类。
- 设计理由：接口一旦不稳定，后面再强的模型能力也无法落地。

### 2.2 路由先于生成

- 先做意图和复杂度判断，再选择 `basic/deep_thinking/web_search`。
- 不是所有请求都走最重链路，复杂请求才进入深度诊断和审计。
- 设计理由：把时延和成本用在真正需要的地方。

### 2.3 证据先于观点

- 诊断结果强制拆成 `facts/hypotheses`。
- `facts` 必须可指向输入上下文；`hypotheses` 必须给验证路径。
- 设计理由：团队协作排障时，能快速区分“已知”与“待验证”。

### 2.4 安全策略是硬门，不是提示词

- Anchor Guard：缺关键锚点时，不输出高风险可执行代码。
- Quality Gate：语法与危险模式检查，按风险级别降级。
- 设计理由：高风险输出不能依赖“模型自己谨慎”。

### 2.5 可靠性是默认配置，不是上线后补丁

- 会话存储采用 SQLite，可选接入 Qdrant 做长期记忆。
- 默认配套 WAL、重试、限流、日志和备份。
- 设计理由：先保证可运行、可排障，再谈能力上限。

## 3. 可交付能力和对应业务价值

| 已实现能力 | 工程问题 | 业务侧结果 |
|---|---|---|
| 多模式路由（`basic/deep_thinking/web_search`） | 所有请求同一处理路径导致波动 | 不同问题走不同链路，稳定性更高 |
| 结构化诊断（`facts/hypotheses`） | 诊断过程不可复核 | 可做团队评审，能逐条验证 |
| 二阶段审计（second pass） | 单次生成偏差难发现 | 关键结论有二次挑战机制 |
| Anchor Guard | 信息不全仍给可执行方案 | 自动降级为安全建议/伪代码 |
| Quality Gate | 代码表面可读但存在风险 | 输出可分级拦截，降低误用 |
| SSE 契约化事件 | 前端状态处理不稳定 | 可编程消费，联调成本下降 |
| SQLite/Qdrant 记忆 | 长会话上下文丢失 | 上下文延续和追踪能力提升 |
| 限流/日志/备份 | 故障难定位、难恢复 | 有基本运行保护和复盘依据 |

## 4. 设计原则是可检查的约束

### 4.1 Failure-First

先定义失败场景，再设计成功路径。  
失败场景至少包括：证据不足、上下文缺失、输出风险过高。

### 4.2 Boundary-First

边界写在流程里，不写在口号里：

- 允许输出可执行代码的前置条件。
- 必须降级为思路/验证步骤的触发条件。
- 可以进入最终正文的信息范围。

### 4.3 Evidence-First

证据不够就明确“不确定”，不把推测包装成结论。  
对生产场景，错误的确定性比明确的不确定性更危险。

## 5. 客户能直接感知的收益

1. 排障效率更高：诊断信息结构化，便于多人并行验证。
2. 集成风险更低：事件契约固定，前端无需猜测状态。
3. 上线安全性更好：高风险输出有硬门和降级路径。
4. 复盘成本更低：日志与状态流可追踪具体失败点。

## 6. 当前边界和承诺范围

1. 不承诺“永远正确”，承诺“可观察、可复核、可降级”。
2. 不默认执行本地高风险动作。
3. 公开版本不包含私有执行策略与内部规则细节。
4. 行业复杂场景仍需要业务规则和私有数据联合约束。

## 7. 适配场景

适合：

- 已有研发/运维团队，重视排障质量和稳定交付。
- 需要把模型输出作为系统能力接入，而不是只做演示。
- 对可审计、可追踪、可控风险有明确要求。

不适合：

- 只追求一次性 Demo，不关心治理和维护成本。
- 希望系统默认执行高风险操作且不做人审。
