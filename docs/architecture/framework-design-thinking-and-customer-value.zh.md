# 框架设计、思考方式与客户价值映射（实事求是版）

## 1. 设计出发点

我把智能体系统当作一个“可控的软件系统”，而不是一个“会聊天的黑盒”。
核心目标不是让回答听起来更聪明，而是让输出在工程上更可用、更可审计、更可复盘。

设计优先级：

1. 先可控，再智能
2. 先证据，再结论
3. 先可降级，再追求完美回答

## 2. 整体框架（分层视角）

### 2.1 交互与协议层

- 提供同步与 SSE 流式接口
- 对事件进行明确分类：`status/content/final/error`
- 目标：前端和下游服务可以稳定消费，不依赖“猜测模型输出格式”

### 2.2 认知与路由层

- 先做理解与意图判断，再决定是否进入诊断、是否需要代码、是否触发二阶段审计
- 目标：让不同请求走不同路径，避免“一把梭”式回答

### 2.3 诊断与审计层

- Evidence-First Diagnosis：输出结构化事实和假设
- Second-Pass Audit：对草稿进行独立复核并有条件合并
- 目标：减少“看起来合理但没有证据”的答案

### 2.4 安全收口层

- Anchor Guard：锚点不完整时阻断高风险可执行代码
- Code Quality Gate：语法/危险模式检查与分级降级
- 目标：降低错误代码直接被复制进生产的风险

### 2.5 记忆与可靠性层

- SQLite 会话存储 + 可选 Qdrant 长期记忆
- WAL、重试、日志、限流、可选备份
- 目标：保障可持续运行与问题排查

## 3. 我目前已经实现的能力与客户价值

| 已实现能力 | 对应客户痛点 | 可交付价值 |
|---|---|---|
| 多模式对话（basic/deep/web） | 所有问题都走同一策略，回答质量波动大 | 让不同复杂度问题使用不同处理路径 |
| 结构化诊断（facts/hypotheses） | 故障分析缺少证据链，无法复核 | 产出可检查的诊断结构而不是纯叙述 |
| 二阶段审计（second pass） | 一次生成容易过度自信或漏项 | 对草稿做独立挑战，降低单次偏差 |
| Anchor Guard | 锚点不全仍输出具体代码，落地失败 | 缺上下文时改为安全建议/伪代码 |
| 质量门（quality gate） | 代码看似完整但语法或风险问题多 | 产物分级并支持安全降级输出 |
| SSE 输出契约 | 前后端对接不稳定、状态不可见 | 明确事件边界，降低集成不确定性 |
| 会话与记忆（SQLite/Qdrant） | 长对话上下文丢失、难以追踪 | 提升会话连续性和可回溯性 |
| 限流/日志/备份 | 运行中缺保护，故障难定位 | 提供基础运行稳定性与排障抓手 |

## 4. 我的工程思考方式

### 4.1 失败优先（Failure-First）

先思考“哪里会坏”，再决定“如何变强”。
所以系统里会优先出现：状态机、降级路径、硬边界，而不是堆功能。

### 4.2 明确边界（Boundary-First）

我把以下边界写死在设计里：

- 什么场景可以输出可执行代码
- 什么场景必须退化成伪代码/验证步骤
- 什么输出能进入最终正文，什么只能作为元数据

### 4.3 证据约束（Evidence-First）

如果证据不足，系统应该显式表达不足，而不是“补全一个看上去很完整的答案”。
这对企业场景非常关键，因为错误的确定性比不确定性更危险。

## 5. 客户能获得的真实收益

1. 问题定位更可复核：不仅有结论，还有证据结构和验证步骤
2. 输出更可集成：流式事件有契约，不需要前端猜状态
3. 风险更可控：高风险代码建议在上下文不全时会被抑制
4. 复盘更容易：日志与状态机有利于定位“哪一步出现偏差”

## 6. 当前边界（不会夸大的部分）

1. 这不是全自动执行型 agent，不会默认替客户执行系统级操作
2. 这不保证“永远正确”，但能把错误限制在可观察、可降级的范围
3. 本公开版不包含本地执行层与私有策略细节
4. 复杂行业场景仍需要业务侧规则与数据配合

## 7. 适用客户场景

- 有研发/运维团队，且重视故障分析质量
- 需要把 LLM 回答接入前端或内部系统
- 对“可追踪、可审计、可控风险”有明确要求

不适用场景：

- 只追求一次性 Demo 展示，不关心稳定性与治理
- 希望系统默认执行高风险本地操作
