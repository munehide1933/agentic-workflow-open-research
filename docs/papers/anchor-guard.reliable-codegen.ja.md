# Anchor Guard: 本番志向 AI コード生成の最終安全レイヤー

## 要旨

AI 生成コードは見た目が正しくても、実行環境に適合しないことが多くあります。
主要因は環境アンカー不足です。
本稿は Anchor Guard、Canonical Artifacts、Quality Gate の三層設計を示します。
狙いは「コード量の最大化」ではなく、「不確実性下での安全な実装可能性」です。

## 1. 序論: コード生成のラストマイル問題

現場での失敗は次に集約されます。

- コンパイルは通るが実環境前提が違う
- runtime/SDK 仮定が暗黙
- 危険スニペットがそのまま運用に流入

これは品質問題に留まらず、運用リスクの転嫁です。

## 2. Anchor（アンカー）とは何か

アンカーは、生成物を実行環境へ結び付ける必須コンテキストです。

- runtime
- client SDK
- HTTP client 前提
- deployment context

アンカーなしのスタック固有コードは推測に過ぎません。

## 3. 三重ガード設計

### 3.1 Anchor Guard（ハード熔断）

診断系コード要求でアンカー不備がある場合:

1. スタック固有の実行コードを遮断
2. スタック非依存方針へ収束
3. 不足アンカーを明示

### 3.2 Canonical Artifacts（単一コードソース）

実行可能コードは canonical artifact チャネルに限定します。
本文の散在コードを実行源にしないことで、検証と統合を単純化します。

### 3.3 Quality Gate（Pass / Soft-Fail / Hard-Fail）

- `pass`: 受け入れ可能
- `soft_fail`: 劣化付き許容（注意喚起）
- `hard_fail`: 実行コード禁止、代替出力へ

目的は「常にコードを出すこと」ではなく、リスク露出を制御することです。

## 4. Partial Salvage（部分回収）

安全に完全コードを返せない場合でも、次の価値を返します。

- 最小擬似コード骨格
- スタック比較マトリクス
- 検証手順

これにより、危険な疑似正解を避けながら開発者の進行を止めません。

## 5. 実運用シナリオ

### シナリオ A: Node runtime 不明 + HTTP client 未指定

- リスク: 実装前提が外れたコード
- 対応: Anchor Guard で具体実装を抑止し、多スタック方針を返す

### シナリオ B: 構文は通るが危険パターン検出

- リスク: 運用事故を誘発する処理混入
- 対応: quality gate で降格または遮断し、安全代替を提示

### シナリオ C: 証拠不足の障害診断

- リスク: 過剰断定の修正コード提案
- 対応: 検証優先の提案 + 境界付き擬似コード

## 6. 企業向け論点: 自由対話型コード提案の限界

無制御なコード対話は次を招きます。

- 説明責任の欠落
- 再現性の欠落
- 隠れ前提による高い影響範囲

状態機械 + ガード設計は、実行級助言の前に明確な関門を設けます。

## 7. 公開戦略

実務上有効な公開方針:

- 公開: 契約、状態遷移、ガード方針
- 非公開: プロンプト内部、閾値、ローカル実行層、製品固有 UI 結合

これにより、技術的信頼性は示しつつ商用上の核を保護できます。

## 8. 結論

信頼できる AI コード生成は、主にシステム設計の課題です。
Anchor Guard と関連ゲートは、モデル出力を「もっともらしい文章」から「境界付き工学成果物」へ変換します。
